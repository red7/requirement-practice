Requirement Dojo (MVP)
1. 项目概览
项目名称： Requirement Dojo (需求道场) 核心目标： 一个沉浸式 Web 应用，模拟真实世界中“非完美”的业务沟通，训练用户捕捉核心需求、撰写结构化文档并应用 AI 思维解决问题的能力。 交互逻辑： 类 Gemini 的聊天界面 + 双向联动的工作台。

2. 核心技术栈建议
Frontend: React (Next.js), Tailwind CSS, Lucide React (Icons).

State Management: React Context API 或 Zustand.

Charts: Recharts (用于五维雷达图)。

Backend/AI: Vercel AI SDK (接入 Claude/Gemini 模型)。

3. 核心功能模块与状态机
应用分为五个关键阶段，请按此逻辑构建：

Phase 1: 场景初始化 (Init)
行业选择： 金融、合规、医疗、电商、运动。

角色性格： * 小白级别： 逻辑清晰，表达完整。

职场现实： 表达破碎、带有情绪、存在隐藏约束。

任务生成： 系统基于选择，生成一个简短的“任务背景”（如：医疗挂号系统效率低下）。

Phase 2: 沉浸式访谈 (Chat)
交互体验： 左侧聊天流，AI 扮演业务方。

AI 行为逻辑： * 包含 30% 无关噪音（抱怨、琐事）。

隐藏 2 个核心痛点和 1 个硬性技术约束。只有当用户提问足够具体或结构化时才触发释放。

快捷工具： 聊天内容支持点击“提取至工作台”。

Phase 3: 需求记录 (Documenting)
分屏显示： 左侧保留聊天记录，右侧出现编辑器。

模板内容： 业务目标、痛点清单、核心功能列表。

协作模式： AI 可以辅助生成初稿，但允许用户手动精修。

Phase 4: 方案设计与 AI 创新 (Design)
任务： 用户提交最终功能设计。

强制模块： [AI 结合点挑战]。用户必须说明：如果引入 AI 技术（如 Agent, RAG, CV 等），如何彻底优化当前流程？

Phase 5: 五维复盘 (Review)
雷达图可视化： 基于以下五个维度打分：

需求洞察力 (Insight)： 是否挖出了隐藏约束。

逻辑完备性 (Logic)： 方案是否覆盖了所有已确认的需求。

AI 思维 (AI-First)： AI 结合点是否具备实战价值而非噱头。

文档专业度 (Professionalism)： 表达是否精准、结构化。

方案可行性 (Feasibility)： 技术上是否可以落地。

专家视角： 展示系统生成的“高阶 AI 解决方案”作为对比参考。

4. 关键 AI 逻辑 (System Prompts)
Prompt A: 业务方角色 (The Stakeholder)
你现在是[行业]的业务人员。你的语言风格是口语化的，偶尔会抱怨工作压力。你希望解决[痛点]，但你不知道技术如何实现。除非用户通过结构化提问（例如：追问流程细节或异常场景），否则你不要主动说出[隐藏约束条件]。

Prompt B: 裁判员角色 (The Judge)
对比用户的访谈记录、文档和方案。请严格按照 1-100 分对五个维度（Insight, Logic, AI-First, Professionalism, Feasibility）进行打分。 输出格式要求： 必须返回结构化的 JSON 数据，包含各维度分数、一段犀利的点评以及 3 个更具启发性的 AI 结合建议。

5. UI/UX 风格指南
配色方案： 深色/浅色自适应模式，主色调建议使用深紫色（#6D28D9）或科幻蓝（#2563EB）。

响应式： 优先保证桌面端大屏体验（因为涉及大量文字输入）。

动效： 提交方案后的雷达图生长动画。

6. Claude Code 执行指令 (可直接复制)
"请根据上述 PRD 构建一个名为 'Requirement Dojo' 的 React 应用。

第一步：先搭建基础的 UI 框架，包含 Phase 1 的场景选择界面。

第二步：实现 Phase 2 的聊天引擎，确保 AI 能够根据设定的 Persona 进场。

第三步：集成 recharts 并编写 Phase 5 的评估逻辑，确保评分能以雷达图形式正确渲染。

注意：代码风格要简洁，组件化，逻辑层和 UI 层分离。"
